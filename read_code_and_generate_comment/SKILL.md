## 目的

为现有项目源码补充**便于学习和阅读的中文注释**，帮助理解每一段有效逻辑的作用，同时避免无意义、冗余或噪声过高的逐词翻译式注释。

## 适用场景

在以下场景触发本 skill：

- 用户要求“逐行阅读代码并添加中文注释”。
- 用户希望快速看懂某个模块、目录或类的作用。
- 用户要求对现有源码做**学习型注释增强**，而不是功能改造。
- 用户明确说明需要中文解释每行代码、每段逻辑或每个关键分支的含义。

## 执行原则

遵循以下总原则：

1. **先理解，再注释**：先完整阅读文件或目标范围，再下手修改，避免边看边猜。
2. **解释"作用"而非翻译"语法"**：说明这行代码为什么存在、完成什么目的，不要把 Java 关键字直译成中文废话。
3. **只给有信息量的代码加注释**：对简单、重复、显而易见的语句保持克制。
4. **不改变代码行为**：仅添加注释，不顺手重构、重排格式或改逻辑，除非用户额外要求。
5. **保持注释简短准确**：优先使用简洁中文，避免长段解释淹没源码。
6. **优先写在语句上方**：尽量把中文注释放在对应语句、分支或逻辑块的上一行，减少对原有代码结构的干扰。

## 强制规则

严格执行以下规则：

1. `import`、`package` 行**不添加注释**。
2. **单元测试代码不添加注释**。
   - 包括测试目录下的文件。
   - 包括名称明显为测试用途的类与方法，例如包含 `Test`、`ITCase`、`TestBase` 等。
   - 如果文件主体是测试代码，则整个文件跳过。
3. **函数体代码不超过 5 行时，不逐行添加注释**。
   - 只需要在函数定义上方补充"函数作用"注释。
   - 如果函数原本已有清晰注释，仅在确有必要时补充。
4. **变量声明和变量引用行不添加注释**。
   - 例如单纯的局部变量定义、字段赋值、getter 返回字段、把变量作为参数传递等。
   - 如果变量声明所在语句同时承载关键业务意图，可只对整段逻辑做概括性注释，不解释变量名本身。
5. **极其简单的算数运算逻辑不添加注释**。
   - 例如简单加减乘除、自增自减、基础索引偏移等显而易见表达式。
6. **文件处理完成后添加完成标记**：当对某个文件处理完毕时，在该文件的最前面加上一行注释，注释的文字是：`// 这个文件已经全部加上中文注释`（对于Scala/Java文件使用`//`，其他语言使用对应的单行注释语法）。
7. **跳过已标记文件**：处理某个文件时，如果发现文件开头已有完成标记注释（`// 这个文件已经全部加上中文注释`），则跳过该文件，不再重复添加中文注释。

## 注释决策流程

按以下顺序判断是否添加注释：

### 1. 识别是否应跳过整个文件

执行以下检查：

- 判断是否属于测试源码。
- 判断用户是否明确排除该文件。
- 判断文件是否主要由生成代码、样板代码或外部依赖代码组成；如果是，优先谨慎处理并遵守用户范围要求。
- 检查文件开头是否已有完成标记注释（`// 这个文件已经全部加上中文注释`），如果有则跳过该文件。

### 2. 识别适合加注释的对象

优先处理以下内容：

- 类、接口、枚举的职责。
- 公开方法、核心私有方法的作用。
- 条件分支的判断意图。
- 循环的遍历目标与退出条件。
- 异常处理、兜底逻辑、状态切换。
- 数据转换、序列化、反序列化、校验、资源管理等关键逻辑。
- 多步链式调用背后的整体目的。

### 3. 识别应跳过的细粒度语句

以下内容默认跳过，不单独写注释：

- `package` / `import`。
- 单纯变量声明或变量传递。
- 简单 setter / getter。
- 显而易见的常量返回。
- 极简单算数表达式。
- 仅仅调用一个命名已非常清晰的方法，且上下文无额外业务信息时。
- 如果某行代码或函数上方已有中文注释，则跳过该行或函数。

### 4. 决定注释粒度

按"最少但足够理解"的原则选择粒度：

- 对短小清晰的方法：只写方法作用注释。
- 对连续多行共同完成一个动作的代码：写一条块级注释即可。
- 对复杂分支、状态机、边界处理：按关键步骤分别注释。
- 不要为了满足"逐行"字面要求而给每个符号、每个变量引用都加注释。

## 推荐输出形式

优先使用以下形式：

### 类或接口说明

在定义上方补充一句或两句中文注释，概括职责。

### 方法说明

对方法补充中文注释，重点说明：

- 方法要解决什么问题。
- 关键输入或输出的语义。
- 必要时说明副作用、边界条件或异常语义。

### 语句或逻辑块说明

在对应代码上方补充单行中文注释，例如：

- 说明本分支为何成立。
- 说明此处为何提前返回。
- 说明此处为何缓存、复制、关闭、校验或转换。

## 操作步骤

执行任务时遵循以下步骤：

1. 明确用户要求的目录、模块或文件范围。
2. 搜索并排除测试代码。
3. 检查目标文件是否已有完成标记注释（`// 这个文件已经全部加上中文注释`），如果有则跳过该文件。
4. 先完整阅读目标文件，再决定注释点位。
5. 优先为类型定义、方法定义和关键逻辑块添加中文注释。
6. 跳过用户明确要求不注释的语句类型，以及已有中文注释的代码行或函数。
7. 完成修改后，快速复查注释是否准确、克制、无明显噪声。
8. 在处理完成的文件开头添加完成标记注释（`// 这个文件已经全部加上中文注释`）。
9. 如有可用 lint 或编译检查，只在必要范围内验证未引入语法问题。

## 质量标准

确保生成的注释满足以下标准：

- **准确**：和真实逻辑一致，不凭空猜测。
- **简洁**：一句话能说清就不要写两句。
- **有用**：帮助理解设计意图，而不是复述语法。
- **一致**：同一文件内语气、粒度和风格保持统一。
- **低侵入**：尽量不打乱源码布局。

## 处理特殊情况

遇到以下情况时这样处理：

- **方法很短但含义不明显**：只补方法注释，不逐行解释。
- **一行代码很长但本质只是变量装配**：不逐项解释，可在整个构造逻辑前写一条概括注释。
- **链式调用较长**：解释整条调用链最终目的，不解释每个中间方法名。
- **复杂条件表达式**：解释判断依据和业务意图。
- **异常分支**：说明为什么在这里抛错、降级或提前返回。
- **并发、锁、状态切换、资源释放**：优先注释，因为这类逻辑通常最需要帮助理解。
