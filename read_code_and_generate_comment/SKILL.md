## 目的
为现有项目源码补充**便于学习和阅读的中文注释**，帮助理解每一段有效逻辑的设计意图。

## 强制执行规则（高优先级）
1. **当前目录优先原则**：处理任何目录时，必须确保该目录下所有源代码文件处理完毕，方可进入其子目录。永远优先处理当前层级，绝不可跳过当前层级去处理其他模块。
2. **测试文件排除**：严格忽略所有单元测试文件（包含 `Test`、`ITCase` 等关键字或位于 `test` 目录下）。
3. **完成标记跳过**：处理文件前，检查文件首行是否已有 `// 这个文件已经全部加上中文注释`，如有则直接跳过该文件。
4. **单次处理完成标记**：文件处理完毕后，必须在首行添加该标记。

## 执行原则
1. **解释设计意图而非逐行翻译**：禁止机械的语义翻译，重点解释“为什么这样做”。
2. **全局视角，分层注释**：关注类职责、方法目的和复杂逻辑块。
3. **保持低侵入性**：尽量将注释置于相关逻辑块上方，不打乱代码结构。
4. **克制原则**：跳过简单 `import`、`package`、Getter/Setter、简单变量声明及显而易见的算术表达式。

## 注释决策流程
1. **处理粒度**：
   - 对类/方法：补充职责或功能说明。
   - 对复杂分支/状态切换/循环：说明判断依据和意图。
   - 对简单函数：若函数体 < 5 行，仅在定义上方补充一句概括。
2. **逻辑块说明**：使用块级注释替代繁琐的逐行翻译。
3. **特殊情形处理**：
   - 链式调用：解释整体目的。
   - 异常分支：说明抛错/降级的原因。
   - 关键资源：锁、并发控制、资源释放必须详细注释。

## 操作步骤
1. **目录递归检查**：递归处理时，先处理当前目录所有源码文件（按字典序），再进入子目录。
2. **状态记录**：使用 `code_comment_done.conf` 记录进度以防重复。
3. **质量复查**：完成修改后检查注释的准确性与克制度。
